# molmanager � architecture notes (v1.4 update log)

## Refactors applied (tracked from holistic review)

- **`dialogs` package**: Replaced monolithic `molmanager/ui/dialogs.py` with `molmanager/ui/dialogs/` (`scope`, `render_2d`, `properties`, `calculator`, `fp_similarity`, `pka`, `protomer`, `cluster`, `mol_tools`) and `__init__.py` re-exports. Imports such as `from molmanager.ui.dialogs import ClusterDialog` are unchanged.
- **`display_constants.py`**: Structure column 2D layout defaults live in `molmanager/display_constants.py`. `compound_table_model` imports from there and still re-exports the same names. `workers/load_render.py` imports layout sizes from `display_constants` (no worker ? `ui.compound_table_model` dependency for those constants).
- **`ClusterMixin`**: Cluster dialog wiring and `on_cluster_*` handlers live in `molmanager/ui/main_window/cluster_mixin.py`; `ChemicalTableApp` includes `ClusterMixin` alongside `ChemistryMixin`.
- **`rdkit_env.py`**: `configure_rdkit_for_desktop_app()` runs from `app.py` after logging setup; duplicate `RDLogger.DisableLog("rdApp.*")` calls were removed from mixins / `chemical_table_app`.
- **`services/`**: Package docstring clarified; still a thin placeholder for future non-Qt domain helpers.

---

## What is working well

**Process and threading model**  
You separate a general `QThreadPool`, a render-only pool, and a single-thread `ProcessQueueManager` for long tool jobs with cooperative cancel. That is coherent and avoids a lot of classic �everything on one pool� bugs.

**Table model direction**  
`CompoundTableModel` already uses batched `dataChanged`, contiguous runs for column updates, and incremental invalidation for numeric bounds. That is the right place for table performance and matches Qt guidance.

**Configuration as policy**  
`config.py` centralizes debounce thresholds, pool sizes, SQL limits, and logging. That makes behavior tunable without hunting through the UI.

**Tests exist**  
Having `tests/` cover model, workers, filters, and session material is a real strength for refactors later.

## Broad structural issues (remaining or softened)

**1. `ChemistryMixin` as a large coordination surface**  
Still roughly 1.5k+ lines in one mixin that owns most tools, enqueue, completion handlers, and status/progress. It remains the main **cohesion bottleneck** for new chemistry features. `.cursor` rules on avoiding god objects still apply here in spirit (god *coordination* layer).

**2. ~~`dialogs.py` as a mega-module~~ ? addressed**  
Dialogs are split by area under `ui/dialogs/`. Further wins: keep new dialogs in small modules; avoid re-growing a single �kitchen sink� file.

**3. ~~Layer bleed (workers ? UI for constants)~~ ? addressed for 2D sizes**  
`load_render` no longer imports `compound_table_model` for depiction width/height. Watch for any new worker imports from `ui.*` for non-UI reasons�prefer `display_constants` or `config` for shared policy.

**4. Mixin implicit contracts**  
`FilterPanelMixin` and friends still assume many attributes on `self`. Flexible, but the contract is **implicit**�easy to break when refactoring or adding a second window type. Optional hardening: typed `Protocol` or a small `TableHost` facade.

**5. ~~Cross-cutting `RDLogger` duplication~~ ? addressed**  
Centralized via `rdkit_env.py` + `app.py`.

## Performance and efficiency (prioritized)

**Already in good shape**  
Incremental load batches, filter debouncing for large tables, batch 2D render via subprocess pool, batched model updates�these are the right levers.

**Worth watching (not necessarily changing now)**  

- **Clustering / fingerprints**  
  Butina, Jarvis�Patrick, and exploratory sweeps are inherently **O(n�)** in pairwise Tanimoto work. For large tables, the right product answer is often **scope** (�selected rows only�), **cap n**, or a future **FPSimilarityTree**-style approximate NN�not micro-optimizing Python loops.

- **Repeated full-table scans**  
  Any new feature that does `for r in range(rowCount)` per keystroke or per filter tick will dominate cost. Prefer pushing work into the model (indexes, dirty flags) or reusing existing debounced filter paths.

- **Singleton / global model state**  
  Things like a shared `QueryModel` for pKa (if still global) reduce load time but can complicate testing and cancellation semantics. Document intent; consider explicit lifetime tied to app or session.

**Legibility vs performance**  
You generally chose clarity (mixins, explicit signal names) over micro-optimizations. For a desktop cheminformatics app, that is appropriate unless profiling shows a hotspot.

## Coherence (how the project �reads� to a newcomer)

**Strengths**  
Clear package split: `ui/`, `workers/`, `config.py`, `app.py`. Naming like `ProcessQueueManager`, `CompoundTableModel`, `ChemicalTableApp` reads well.

**Friction points**  
- **`services/`** is still mostly aspirational�grow it when you have a clear non-Qt boundary to enforce.  
- **Signal ownership** remains mixed (`WorkerSignals` vs dialog-owned vs per-feature `*Signals`)�document a rule of thumb in `docs/` or `CONTRIBUTING` only if the team wants that written down.

---

## Summary judgment

There is **no single �wrong� architecture** here: conventional PyQt + RDKit desktop app with thoughtful batching and threading. The **main remaining structural tension** is concentration of behavior in **`chemistry_mixin.py`** (not emergencies; iterate with small mixins or service modules as features land).

**Done from the earlier follow-up list:** (1) split dialogs by feature area, (2) extract shared layout constants for non-UI consumers, (3) peel clustering into `ClusterMixin`.

**Natural next steps when ready:** thin slices out of `ChemistryMixin` (e.g. conformers-only mixin), typed host protocols for filter/table contracts, and expanding `services/` with real session/file helpers once boundaries are clear.
